## 3. Agents

### Crew 2 — Value Design

#### Value Proposition Generator (Claude Opus 4.6)
- **Interaction:** Automated synthesis → human review gate
- **Tools:** `SQLiteStateTool`, `HumanInputTool`
- **Inputs:** Discovery outputs from SQLite `agent_outputs`:
  - Value chain (pain points and opportunities per cell)
  - Requirements register
  - Value lever register
  - User journey register (if present — detected by querying `agent_outputs` for crew `discovery_journeys`)
- **Process:** Synthesises inputs into a list of value proposition records. Each record contains:
  - `title` — short label
  - `change_articulation` — 2–3 sentence description of the change
  - `impacted_stakeholder_groups` — list from project config
  - `value_estimate` — qualitative sizing: High / Medium / Low with justification
  - `supporting_evidence` — list of source references (value lever, requirement, journey, or pain point IDs)
- **Outputs:** Structured JSON written to `agent_outputs` (crew: `value_design`, agent: `value_proposition_generator`). `human_reviews` record created to gate Portfolio Manager.

#### Portfolio Manager (Claude Haiku 4.5)
- **Interaction:** HITL — polls SQLite for ranking criteria; defers Chainlit to SP4
- **Tools:** `SQLiteStateTool`, `HumanInputTool`, `ExcelOutputTool`
- **HITL flow:** Creates a `human_reviews` record requesting ranking criteria (weights for: value, feasibility, strategic fit). Human submits weights via `PATCH /projects/{slug}/reviews/{id}` with weights in `reviewer_notes` as JSON. Agent polls until response received.
- **Process:** Applies weighted scoring to each proposition. Scores normalised to 0–100. Portfolio ranked by total weighted score.
- **Outputs:**
  - Scored portfolio register → `agent_outputs` (JSON)
  - `portfolio_register.xlsx` → `projects/{slug}/outputs/` via `ExcelOutputTool`
  - Final `human_reviews` gate record to confirm portfolio approved

### Crew 3 — Architecture

#### Enterprise Architect (Claude Sonnet 4.6)
- **Interaction:** Automated extraction → human review gate
- **Tools:** `ChromaQueryTool`, `MermaidRenderTool`, `SQLiteStateTool`, `HumanInputTool`
- **Inputs:** Architecture documents ingested into ChromaDB via existing upload pipeline (org charts, system inventories, technology registers, process documentation)
- **Process:** Queries ChromaDB for the project collection. Extracts and structures a current-state architecture register across three layers:
  - **Data layer** — key data entities, sources, flows, ownership
  - **Technology layer** — systems, platforms, integrations, infrastructure
  - **Organisation layer** — teams, roles, reporting lines, key capabilities
  - One Mermaid diagram generated per layer
- **Outputs:** Architecture register (three-layer JSON) + three Mermaid diagram strings written to `agent_outputs`. `human_reviews` gate record created.

#### Initiative Identifier (Claude Sonnet 4.6)
- **Interaction:** Automated gap analysis → human review gate
- **Tools:** `SQLiteStateTool`, `HumanInputTool`
- **Inputs (all from SQLite `agent_outputs`):**
  - Value propositions (crew: `value_design`, agent: `value_proposition_generator`)
  - Architecture register (crew: `architecture`, agent: `enterprise_architect`)
  - Requirements register (crew: `discovery`, agent: `requirements_analyst`)
- **Process:** For each value proposition, identifies:
  - Capability gaps — what the proposition requires vs. what the architecture currently provides
  - Initiative category: `enabling` / `operating_model` / `business_change`
  - Complexity score: 1 (simple) to 5 (transformational) with rationale
- **Outputs:** Initiative register (list of initiative records with gap analysis, category, complexity) written to `agent_outputs`. `human_reviews` gate record created.
- **Note:** Initiative Identifier reads Value Design outputs from SQLite, so it naturally runs after Value Design completes even when crews are triggered independently — no orchestration logic required in SP3b.

## 4. New Tool: ExcelOutputTool

**File:** `agents/tools/excel_output.py`  
**Pattern:** `BaseTool` subclass, `_run(self, **kwargs)`, Pydantic input schema. Registered in `registry.py` as `ExcelOutputTool`.

**Input schema:**
- `rows` — list of dicts (uniform keys)
- `columns` — ordered list of column name strings
- `filename` — output filename (e.g. `portfolio_register.xlsx`)
- `project_slug` — used to resolve output path: `projects/{slug}/outputs/{filename}`

**Behaviour:**
- Writes XLSX using openpyxl
- Bold header row, auto-width columns, freeze top row
- Returns absolute file path

**Used by:** Portfolio Manager

## 6. HITL Pattern

Identical to SP3a. No Chainlit in SP3b.

1. Agent creates a record in `human_reviews` with `status = pending` and a `prompt` field describing what is needed
2. Human calls `PATCH /projects/{slug}/reviews/{id}` with `{"status": "approved", "reviewer_notes": "..."}` (or `"rejected"`)
3. Agent polls SQLite via `HumanInputTool` until status changes
4. On approval, agent reads `reviewer_notes` and continues; on rejection, agent revises and creates a new review record
