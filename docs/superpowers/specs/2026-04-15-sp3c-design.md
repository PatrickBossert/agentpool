## 3. Agent — Roadmap Generator

### Identity
- **Role:** `"Roadmap Generator"`
- **Model:** Claude Sonnet 4.6 — `get_crew_llm(llm_mode)`
- **Tools:** `SQLiteStateTool`, `HtmlRoadmapTool`, `HumanInputTool`
- **allow_delegation:** False

### Inputs
All read via `SQLiteStateTool`:
- `key='initiative_register'` — SP3b Architecture crew output
- `key='propositions'` — SP3b Value Design crew output
- `key='value_levers'` — SP3a Discovery crew output

Config values embedded in the task description string at crew assembly time:
- `value_stream_labels` — defines the value stream row groups
- `stakeholder_groups` — defines the per-group rows within each value stream block
- `roadmap_time_axis` — `"quarters"` | `"years"` | `"horizons"`

### Process (task description steps)

1. Read `key='initiative_register'` via SQLiteStateTool.
2. Read `key='propositions'` via SQLiteStateTool.
3. Read `key='value_levers'` via SQLiteStateTool.
4. Sequence initiatives into named time periods using LLM reasoning:
   - Enabling initiatives first (lower complexity_score first within category)
   - Then operating_model initiatives
   - Then business_change initiatives
   - Name periods from the configured time axis (e.g. "Q1 2026", "Q2 2026" for quarters; "Year 1", "Year 2" for years; "Horizon 1", "Horizon 2", "Horizon 3" for horizons)
   - Generate as many periods as needed to fit all initiatives; minimum 3.
5. For each initiative, assign:
   - `period` — sequenced time period
   - `value_streams` — value stream labels this initiative primarily supports (derived from its `proposition_ids` and `category`)
6. For each value proposition, assign:
   - `realisation_period` — the period when its last enabling initiative completes (i.e. benefits are realised after the enabling work is done)
   - `value_streams` — value stream labels (from `impacted_stakeholder_groups` mapped to `value_stream_labels`)
   - `value_levers` — lever names traced from `supporting_evidence` lever references back to the value lever register
7. Assemble the complete roadmap JSON object (schema in Section 4).
8. Write via `SQLiteStateTool` with `operation='write'`, `key='roadmap_data'`, `agent_name='roadmap_generator'` — this produces `outputs/roadmap_data.json`.
9. Call `HtmlRoadmapTool` with the assembled roadmap object, `filename='roadmap.html'`, `agent_name='roadmap_generator'` — produces `outputs/roadmap.html`.
10. Call `HumanInputTool` with prompt: `'Please review the roadmap at outputs/roadmap.html and the underlying data at outputs/roadmap_data.json. Reply "approved" to conclude Delivery Planning, or provide revision notes.'`
11. If revision notes received, revise the roadmap and call steps 8–10 again. Maximum 3 revision cycles.

### Outputs
- `outputs/roadmap_data.json` — via SQLiteStateTool (key='roadmap_data')
- `outputs/roadmap.html` — via HtmlRoadmapTool

The agent writes this structure to `key='roadmap_data'`:

```json

  "time_axis": "quarters",
  "periods": ["Q1 2026", "Q2 2026", "Q3 2026", "Q4 2026"],
  "value_streams": ["Operations", "IT", "Finance"],
  "stakeholder_groups": ["Investor", "Customer", "Sales", "Operations", "IT", "Finance"],
  "initiatives": [
    {
      "id": "INIT-001",
      "title": "Automate Order Entry",
      "category": "enabling",
      "complexity_score": 2,
      "period": "Q1 2026",
      "value_streams": ["Operations"],
      "proposition_ids": ["VP-001"]
    }
  ],
  "propositions": [
    {
      "id": "VP-001",
      "title": "Automated Order Management",
      "value_estimate": "High",
      "change_articulation": "Replace manual order entry with an end-to-end automated platform.",

      "impacted_stakeholder_groups": ["Investor", "Customer", "Operations"],
      "value_levers": ["Process Automation", "OpEx Reduction"]
    }
  ]

## 5. New Tool — HtmlRoadmapTool

**File:** `agents/tools/html_roadmap.py`
**Pattern:** `BaseTool` subclass identical to `ExcelOutputTool` — Pydantic input schema + `_run()` + `insert_agent_output_sync`.

### Input Schema
- `roadmap_data` — dict (the full roadmap JSON object as defined in Section 4)
- `filename` — output filename (e.g. `roadmap.html`)
- `agent_name` — for output tracking

### Behaviour
- Writes a self-contained single HTML file (inline CSS, no external JS dependencies) to `projects/{slug}/outputs/{filename}`
- Calls `insert_agent_output_sync` with `output_type="html"`
- Returns the absolute file path on success; `"Error: render failed — {reason}"` on failure

### HTML Grid Structure

For each value stream in `roadmap_data.value_streams`, renders a labelled section containing:

| Row type | Content |
|---|---|
| **One row per stakeholder group** (from `stakeholder_groups` config, embedded in roadmap_data) | VP title shown in the `realisation_period` column for each group listed in the VP's `impacted_stakeholder_groups` |
| **Capability Builds** | Initiative titles in their `period` column, coloured by category: enabling = blue, operating_model = amber, business_change = green; complexity score shown as a small badge |
| **Benefits** | Value lever names (e.g. `Process Automation · OpEx Reduction`) + value estimate badge (`High` / `Medium` / `Low`) in the `realisation_period` column |

Column headers are the period names. All value stream sections share the same column grid.

### Example rendered structure (Operations value stream)

```
┌─────────────────────────┬──────────────┬──────────────┬──────────────┐
│ OPERATIONS              │   Q1 2026    │   Q2 2026    │   Q3 2026    │

│ Investor                │              │ Auto. Order  │              │
│ Customer                │              │ Auto. Order  │              │
│ Operations              │              │ Auto. Order  │ Int. Supply  │
│ IT                      │              │              │ Int. Supply  │
├─────────────────────────┼──────────────┼──────────────┼──────────────┤

├─────────────────────────┼──────────────┼──────────────┼──────────────┤

│                         │              │ OpEx · HIGH  │ Visibility · │

## 6. Data Flow

```
SP3a output (SQLite agent_outputs)
    └── key='value_levers'      (Discovery crew)

SP3b outputs (SQLite agent_outputs)
    ├── key='initiative_register'  (Architecture crew)
    └── key='propositions'         (Value Design crew)
            │
            ▼
    Roadmap Generator
            │  reads: initiative_register, propositions, value_levers
            │  sequences initiatives into periods (LLM reasoning)
            │  assigns realisation_period to each VP
            │  traces value lever names from supporting_evidence
            │  writes: key='roadmap_data' → outputs/roadmap_data.json
            │  calls: HtmlRoadmapTool → outputs/roadmap.html
            │  creates: human_reviews gate record
            ▼
    [human reviews outputs/roadmap.html]
    [PATCH /projects/{slug}/reviews/{id} → {"status": "approved"}]
            │
            ▼
    [Delivery Planning complete — ready for SP4 Business Plan Generator]
```

## 7. HITL Pattern

Identical to SP3a/SP3b. No Chainlit in SP3c.

1. Agent creates `human_reviews` record with `status = pending` and prompt describing what to review
2. Human calls `PATCH /projects/{slug}/reviews/{id}` with `{"status": "approved", "reviewer_notes": "..."}`
3. Agent polls SQLite via `HumanInputTool` until status changes
4. On approval, agent continues; on rejection, agent revises and creates a new review record
5. Maximum 3 revision cycles
